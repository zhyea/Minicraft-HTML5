## 「底下的入口始终进不去」诊断（纯答疑，未改代码）
- 用户问地表下楼梯入口为何走不进去。
- 排查：逐层审 Player.tick 楼梯检测 → Game.changeLevel(pendingLevelChange) → Game.tick openTransition → TransitionMenu done → completeLevelChange。逐项排除：① stairs tile mayPass 默认 true（可踩上）；② Player.game 引用 = App 的 game 实例（startNewGame 与 SaveManager.EntityIO.read 均正确传 game）；③ onStairDelay 初值 0；④ 方向正确（stairsDown→dir-1→currentLevel 3→2）；⑤ 读档路径 player.game 也正确。
- **根因（地形生成，非换层 bug）**：LevelGen 生成地表 `stairsDown` 的硬规则要求周围 8 格全为 Rock（否则不放）；Rock.mayPass=false（实心）。故下楼入口被一圈岩石墙包死，玩家到不了楼梯格中心 → 换层不触发。同理，地表去天空的 stairsUp 周围被 HardRock 围（Level.ts 77-88 行，surface depth===0 → 设 hardRock），硬岩仅宝石镐(level4)可挖。
- 这是忠实移植原版 Minicraft 的设计（原版地表楼梯即被岩石围，需挖开进出）；对新手不直觉，像 bug。
- 验证：新增集成测试 `ts-port/src/game/__tests__/stairs-levelchange.test.ts`（1 测试：teleport 玩家到 surface stairsDown → tick → currentMenu='transition'、levelChangeDir=-1 → completeLevelChange → currentLevel 3→2、玩家搬至 UG1）。实测通过：换层机制 100% 正确，坐实"进不去"是可达性问题而非逻辑 bug。vitest 118（含新增 1）通过。未 git commit。
- 给用户的方案：① 保持原版——拿镐挖掉楼梯周围岩石（Rock 任意级镐可挖）/硬岩（需宝石镐）再走上楼梯格触发过场换层；② 改 LevelGen 让楼梯周围留可走地面（牺牲 fidelity 换可达性）。**用户拍板：保持原版设计（不改代码），靠挖岩石/硬岩进出。**

## 「吃苹果弹出问号」诊断（纯答疑，未改代码）
- 用户问吃苹果时弹出的问号是什么。
- 排查链路：Player.attack() → activeItem.interactOn(tile) → FoodResource.interactOn（heal 条件：health<maxHealth 且 payStamina 成功）→ Player.feedbackFood(item, ate) 弹 TextParticle。
- feedbackFood 三种反馈：① 成功→绿色 `+N`（N=heal，走 Font 正常渲染）；② 满血→黄色 `吃饱了`；③ 体力不足→黄色 `体力不足`。
- **根因**：游戏世界内的文字粒子走 `TextParticle.render()` → `Font.draw()`（`ts-port/src/engine/Font.ts`）。该字库 `chars` 仅含 A–Z、0–9 与少量标点（`. , ! ? ' " - + = / \ % ( ) < > : ;`），**完全无中文**；`FALLBACK` 表也无汉字，故任意中文字符 `?? FALLBACK['?']` 落回问号。结果 `吃饱了`/`体力不足` 被渲染成一串 `?`。
- 关键推论：成功吃弹的 `+N` 中 `+`(在 chars)与数字均正常→**不出现问号**；用户看到问号=本次吃未成功（满血或体力不足）。
- 注意：背包/菜单中文（木剑、木斧等）走 Vue/DOM 文本，原生渲染正常；仅**游戏世界内粒子**走 canvas 字体才 fallback 成问号。GWT(Java) 版 Font.java 有 CJK 8×8 栅格化，ts-port 的 Font.ts 未搬这套（仅搬了 chars 常数），故 ts-port 世界内中文一律 `?`。
- 待修方向（用户尚未要求）：A. 把那几条提示改英文/符号（FULL / NO STAMINA / +N），零成本贴合像素风；B. 给 ts-port 世界字体补 CJK 运行时栅格化（同 GWT），一劳永逸。未 git commit。
